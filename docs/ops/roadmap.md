## Active blockers and decisions

### Blocker: private-server smoke needs local secret/config setup

Docker/Compose is available, so the infrastructure blocker is resolved. Smoke execution is now blocked only on safe local private-server inputs:

1. provide `STEAM_KEY` or equivalent launcher config outside git;
2. keep private-server config/admin credentials untracked;
3. run the Dockerized smoke runbook and record redacted findings.

Recommendation: execute private-server smoke as soon as credentials/config are available; otherwise continue deterministic hardening that reduces smoke-test risk.

### Decision: next code priority

Recommended next code slice if private-server smoke remains credential-blocked: emergency/runtime hardening for zero-creep or low-energy recovery.

Reason:

- Worker replacement planning and telemetry MVP are now complete.
- Additional recovery hardening can be validated deterministically without private-server credentials.
- It reduces risk before the first real runtime smoke.

### Decision: MMO deployment gate

Recommendation: require private-server smoke before Screeps MMO deployment unless the owner explicitly overrides the risk.

Reason:

- The bot has deterministic test coverage but has not yet run in a real Screeps runtime loop.
- A private smoke should catch packaging/runtime/environment issues before live deployment.

## Main-agent/subagent completion reporting rule

Whenever a subagent finishes a task, the main agent must:

1. read and assess the subagent result before treating it as accepted;
2. identify whether the result changes roadmap, tasks, research findings, implementation state, runtime status, or decisions;
3. post the relevant summary to the corresponding Discord channel(s):
   - `#task-queue` — task status, blockers, done criteria, next task;
   - `#dev-log` — implementation/test/build/file/commit results;
   - `#roadmap` — milestone or phase changes;
   - `#research-notes` — new factual findings and source-derived conclusions;
   - `#decisions` — final decision requests or direction-changing tradeoffs;
   - `#runtime-summary` — routine runtime state after a runtime exists;
   - `#runtime-alerts` — exceptions, deadlocks, resource crises, deployment failures, or urgent runtime issues;
4. avoid interrupting the owner for intermediate progress unless a final decision is needed;
5. record durable process state in `docs/process/` when the task is complex, long-running, or needed for future recovery/blog writing.

Cron caveat:

- Scheduled workers may only be able to deliver a single final response to one configured target. In that case, the worker should place labelled sections such as `#task-queue`, `#dev-log`, and `#research-notes` in its final response, and the continuation job should prefer delivery to `#task-queue` unless true per-channel fan-out is explicitly configured.
